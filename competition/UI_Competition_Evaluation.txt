================================================================================
              GRANJUR DEV PORTAL — UI REDESIGN COMPETITION
                       EVALUATION CRITERIA
================================================================================

Platform Requirement: React Native (Expo)
- Must be fully responsive across web, iOS, and Android
- Single codebase via Expo (React Native for Web included)
- Navigation must feel native on mobile and SPA-like on web

Total Score: 100 points (10 points per category)

Data Requirement: All data must be data-driven, not hardcoded.
- Use stubs that mimic API responses where a backend endpoint is not yet available
- All user-facing lists, permissions, and configurations must come from a store
  (local state, context, or equivalent) populated by stub/API responses
- No hardcoded arrays of users, projects, permissions, or repos in components
- See "Permission Model" section at the bottom for the required structure

================================================================================
 PERMISSION MODEL (Required Data Structure)
================================================================================

All permission checks must be data-driven. The user object in the store must
include a permissions structure. Example:

{
  "user_id": "u_123",
  "email": "john@granjur.com",
  "name": "John Doe",
  "permissions": {
    "manage_users": {
      "included": ["*"]
    },
    "manage_projects": {
      "included": ["proj_badar_hms", "proj_quranflow"]
    },
    "view_projects": {
      "included": ["*"]
    },
    "manage_repos": {
      "included": ["*"]
    },
    "manage_user_projects": {
      "included": ["u_456", "u_789"]
    }
  }
}

Rules:
- Each permission key (e.g. "manage_projects") maps to an object
- The "included" array contains IDs this permission applies to
- Use "*" to indicate the permission applies to ALL IDs
- To check if a user can manage project "proj_badar_hms":
    user.permissions.manage_projects.included includes "proj_badar_hms" OR "*"
- To check if a user can manage another user's projects:
    user.permissions.manage_user_projects.included includes that user's ID OR "*"
    AND user.permissions.manage_projects.included includes the project ID OR "*"

This structure must be populated from a stub (simulating an API response) and
used throughout the app to gate features. It must NOT be hardcoded in components.

Permission keys required:
  manage_users          — can view/edit collaborator list
  manage_projects       — can add/remove/edit specific projects
  view_projects         — can view specific projects
  manage_repos          — can add/remove tracked repositories
  manage_user_projects  — can assign projects to other users (by user ID)
  create_issues         — can create GitHub issues
  view_meetings         — can view meetings
  create_meetings       — can create meetings

 GENERAL NOTES
================================================================================

- The app must be built with React Native (Expo) to support web + mobile
- Responsive design is mandatory — every screen must work on both platforms
- Dark/light theme support must be maintained
- Navigation: use a tab bar or drawer on mobile, sidebar/breadcrumbs on web
- All stub data should be isolated in a /stubs or /data directory
- Stubs should return Promises to simulate async API calls
- No feature should require a live backend to demo — stubs must be sufficient
  for the evaluation
- The competition evaluates UI/UX quality, not backend integration
