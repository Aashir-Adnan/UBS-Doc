## 1. نظرة عامة وملخص القيود <a name="overview"></a>

يجب أن تستوفي جميع الحلول القيود غير القابلة للتفاوض التالية المستمدة من وثيقة متطلبات المنتج:

| القيد | المتطلب |
|---|---|
| **TR-1** | المزامنة عبر السحب فقط؛ يُحظر تمامًا أي دفع من السحابة إلى الشبكة المحلية |
| **TR-2** | فترة المزامنة ≥ ساعة واحدة، أو بشكل غير متزامن عبر أحداث الأعمال OXI/OHIP |
| **TR-3** | لا يُسمح باستخدام عنوان IP ثابت أو إعادة توجيه المنافذ في موقع الفندق |
| **SC-1** | تشفير TLS 1.3 لجميع البيانات أثناء النقل |
| **SC-2** | إخفاء أو تجزئة بيانات التعريف الشخصية (PII) قبل مغادرتها البيئة المحلية |
| **SC-3** | تقييد مستخدم قاعدة البيانات التكاملية بصلاحية `SELECT` فقط |
| **SA-1** | إلزامية المصادقة متعددة العوامل (MFA) لجميع المستخدمين الخارجيين |
| **SA-2** | يجب أن تستهدف قراءات أصحاب المصلحة النسخة المتماثلة السحابية للقراءة فقط |

## 2. حلول القراءة فقط <a name="read-only-solutions"></a>

### الحل الأول: وكيل محلي + نفق Cloudflare + نسخة متماثلة سحابية مُدارة

**ملخص الهندسة المعمارية:**  
يتم تثبيت خدمة خفيفة الوزن على نظام Windows/Linux محليًا تقوم باستطلاع قاعدة بيانات Oracle الخاصة بـ OPERA على فترات منتظمة، وإخفاء بيانات التعريف الشخصية محليًا، ثم بث الحمولة للخارج عبر **نفق Cloudflare** (المعروف سابقًا بـ Argo Tunnel). يُنشئ النفق اتصال HTTPS ثابتًا للخارج فقط مع حافة Cloudflare — مما يلغي الحاجة إلى عناوين IP ثابتة أو قواعد جدار الحماية الواردة. تُخزَّن البيانات في قاعدة بيانات سحابية مُدارة (مثل AWS RDS PostgreSQL أو Supabase) تعمل كنسخة متماثلة للقراءة.

```
[OPERA Oracle DB] ──SELECT──► [On-Prem Agent]
                                    │  PII Masking
                                    │  TLS 1.3
                                    ▼
                           [Cloudflare Tunnel]
                                    │ Outbound only
                                    ▼
                        [Cloudflare Edge / Zero Trust]
                                    │
                                    ▼
                         [Cloud Read-Replica DB]
                                    │
                                    ▼
                    [SaaS Application + MFA Auth Layer]
```

**خريطة الامتثال:**
- ✅ TR-1: الوكيل يسحب من Oracle؛ السحابة لا تدفع بيانات للداخل أبدًا
- ✅ TR-2: جدولة عبر Windows Task Scheduler / cron (ساعة فأكثر) أو مُشغَّل بالأحداث
- ✅ TR-3: نفق Cloudflare لا يتطلب إعادة توجيه المنافذ أو IP ثابت
- ✅ SC-1: نفق Cloudflare يفرض معيار TLS 1.3
- ✅ SC-2: إخفاء PII داخل الوكيل قبل أي مغادرة للبيانات من الشبكة المحلية
- ✅ SC-3: مستخدم Oracle مُنشأ بصلاحية `GRANT SELECT` فقط
- ✅ SA-1/SA-2: MFA مُفعَّل على طبقة SaaS؛ النسخة المتماثلة للقراءة فقط

**المكدس التقني المقترح:** Cloudflare Tunnel، AWS RDS / Supabase، وكيل Python أو Node.js، Oracle JDBC/cx_Oracle  
**التعقيد:** ⭐⭐ منخفض-متوسط  
**ملف التكلفة:** منخفض (الطبقة المجانية من Cloudflare Tunnel متاحة)

**المزايا:**
- أقل حاجز للدخول — نفق Cloudflare يوفر طبقة مجانية سخية وبرنامج خفي `cloudflared` موثق جيدًا لأنظمة Windows وLinux
- لا تغييرات شبكية مطلوبة في موقع الفندق؛ لا يحتاج فريق تقنية المعلومات إلى فتح منافذ جدار الحماية أو طلب IP ثابت من مزود الإنترنت
- لوحة تحكم Cloudflare Zero Trust توفر سياسات الوصول والتوجيه المدرك للهوية وسجلات الاتصال جاهزة للاستخدام
- وكيل الاستطلاع عملية بسيطة عديمة الحالة يسهل صيانتها وإعادة نشرها أو إدارة إصداراتها
- توافق واسع مع قواعد البيانات السحابية — يمكن استضافة النسخة المتماثلة لدى أي مزود (AWS، GCP، Azure، Supabase)

**العيوب:**
- يُدخل الاستطلاع تأخرًا متأصلًا في البيانات؛ ستتخلف النسخة المتماثلة عن OPERA بمقدار الفترة المُهيَّأة (ساعة كحد أدنى وفق TR-2)، مما يجعلها غير مناسبة لحالات الاستخدام شبه الفوري
- وكيل المزامنة مكوّن مخصص يتطلب صيانة مستمرة ومعالجة أخطاء وإدارة تغييرات المخطط مع ترقيات OPERA
- تمثل عملية الوكيل الفردية نقطة فشل محتملة ما لم يُنفَّذ آلية مراقبة أو تكرار
- الطبقة المجانية من Cloudflare تفرض حدودًا على الاتصالات والنطاق الترددي قد تكون غير كافية للنشرات متعددة العقارات الكبيرة؛ الطبقات المدفوعة تُضيف تكلفة متكررة
- منطق إخفاء PII يقطن في كود الوكيل، مما يتطلب اختبارًا ومراجعةً دقيقَيْن لضمان عدم وجود ثغرات تسريب

### الحل الثاني: مستمع أحداث OXI/OHIP + AWS EventBridge + مسار بلا خادم

**ملخص الهندسة المعمارية:**  
بدلًا من الاستطلاع المجدوَل، يستفيد هذا الحل من نظام **أحداث الأعمال OXI/OHIP** الأصلي في OPERA. عند حدوث حجز أو تسجيل وصول أو تغيير سعر، يطلق OPERA حدثًا. تلتقط خدمة مستمع محلية هذه الأحداث، وتُطبق إخفاء PII، وتُعيد توجيهها للخارج عبر **AWS PrivateLink** أو relay آمن للويب هوك إلى **AWS EventBridge**. يوجّه EventBridge الأحداث إلى دوال Lambda التي تُغذّي النسخة المتماثلة السحابية في شبه وقت فعلي.

```
[OPERA PMS] ──Business Event──► [On-Prem OXI Listener]
                                         │ PII Mask + TLS 1.3
                                         ▼
                               [Outbound Webhook Relay]
                                         │
                                         ▼
                              [AWS EventBridge / SNS]
                                         │
                                    ┌────┴─────┐
                                    ▼          ▼
                              [Lambda]    [Lambda]
                             (Upsert)    (Audit Log)
                                    │
                                    ▼
                         [Cloud Read-Replica (RDS)]
```

**خريطة الامتثال:**
- ✅ TR-1: المستمع محلي ومُشغَّل بالأحداث؛ لا اتصالات سحابية واردة
- ✅ TR-2: مُشغِّلات أحداث الأعمال غير المتزامنة تستوفي متطلب OXI/OHIP صراحةً
- ✅ TR-3: ويب هوك HTTPS للخارج؛ لا قواعد جدار حماية واردة مطلوبة
- ✅ SC-1: AWS SDK يفرض TLS 1.3
- ✅ SC-2: إزالة PII في مستمع OXI قبل حمولة EventBridge
- ✅ SC-3: مستخدم تكامل Oracle للقراءة فقط؛ OXI يقرأ الأحداث المنشورة فقط

**المكدس التقني المقترح:** Oracle OXI/OHIP، AWS EventBridge، AWS Lambda، RDS Aurora  
**التعقيد:** ⭐⭐⭐ متوسط  
**ملف التكلفة:** متوسط (أسعار AWS بلا خادم، تتناسب مع حجم الأحداث)

**المزايا:**
- يستوفي بشكل أصلي متطلب TR-2 للأحداث غير المتزامنة — تصل البيانات إلى النسخة المتماثلة في ثوانٍ من تغيير حالة OPERA، لا ساعات
- الهندسة المعمارية المُشغَّلة بالأحداث منفصلة بطبيعتها؛ يمكن توسيع مسار السحابة أو تعديله أو استبداله بشكل مستقل عن المستمع المحلي
- النموذج بلا خادم في AWS Lambda يتوسع تلقائيًا مع حجم الأحداث ولا يتكبد تكلفة في أوقات الخمول
- دعم قوائم الرسائل الميتة المدمج في EventBridge / SQS يوفر مرونة ضد الأعطال العابرة دون فقدان البيانات
- فصل دوال Lambda للإضافة والتدقيق يفرض فصلًا نظيفًا للاهتمامات ويُبسّط إعداد تقارير الامتثال

**العيوب:**
- يتطلب تهيئة OXI/OHIP خبرة معتمدة من Oracle؛ تعيينات أحداث الأعمال غير الصحيحة قد تسبب أحداثًا مفقودة أو مكررة وقد تستلزم دعم Oracle
- لا تعرض جميع إصدارات OPERA أو تهيئات النشر المجموعة الكاملة من أحداث الأعمال المطلوبة — يجب التحقق من التوافق لكل عقار
- الارتباط بمزود AWS: مكدس EventBridge + Lambda + RDS غير قابل للنقل؛ الترحيل إلى مزود سحابي آخر يتطلب إعادة بناء جوهرية
- زمن التأخير عند البدء البارد للدوال بلا خادم قد يؤخر معالجة الحدث الأول بعد فترات الخمول، وإن كان عادةً دون ثانية
- تتناسب تكلفة التشغيل مباشرةً مع حجم الأحداث؛ الفندق عالي النشاط (فندق مؤتمرات كبير) قد يولد تكاليف استدعاء Lambda وEventBridge مرتفعة بشكل غير متوقع

### الحل الثالث: وكيل Relay عبر ngrok/Pinggy + قائمة مزامنة مشفرة

**ملخص الهندسة المعمارية:**  
ينشئ وكيل مزامنة محلي نفقًا دائمًا عكسيًا عبر **ngrok** (أو بديل مستضاف ذاتيًا كـ **frp** أو **Pinggy**) لإتاحة نقطة نهاية gRPC/HTTPS محلية حصريًا للواجهة الخلفية لـ SaaS. ترسل الخدمة السحابية إشارة طلب سحب عبر النفق، محفزةً الوكيل لاستعلام Oracle وإخفاء PII وإعادة حمولة البيانات — كل ذلك ضمن جلسة واحدة مُبادَرة من الخارج. قائمة رسائل (مثل Redis Streams أو SQS) تعمل كمخزن مؤقت للتسليم الموثوق.

```
[SaaS Cloud Backend] ──pull signal──► [ngrok Cloud Edge]
                                               │ (outbound tunnel, no port forward)
                                               ▼
                                    [On-Prem ngrok Agent]
                                               │ SELECT query
                                               ▼
                                       [OPERA Oracle DB]
                                               │
                                    [PII Mask → Payload]
                                               │ TLS 1.3 response
                                               ▼
                                    [ngrok Cloud Edge]
                                               │
                                               ▼
                                  [Cloud Queue → Read-Replica]
```

> **ملاحظة:** إشارة السحب تنتقل من السحابة إلى حافة النفق، لكن **اتصال البيانات يُبادَر ويُحافَظ عليه من قِبَل الوكيل المحلي** (للخارج)، مما يُحافظ على امتثال TR-1. السحابة لا تستطيع بدء جلسة TCP جديدة داخل الشبكة المحلية.

**خريطة الامتثال:**
- ✅ TR-1: جلسة TCP تُنشأ دائمًا للخارج من قِبَل الوكيل المحلي
- ✅ TR-2: وتيرة السحب قابلة للتهيئة؛ متوافقة مع المُشغِّلات بالأحداث
- ✅ TR-3: نفق ngrok يلغي متطلب IP ثابت / إعادة توجيه المنافذ
- ✅ SC-1: ngrok يفرض TLS 1.3 على جميع حركة النفق
- ✅ SC-2: إخفاء PII قبل أن تغادر الحمولة الوكيل
- ✅ SC-3: بيانات اعتماد Oracle للقراءة فقط

**المكدس التقني المقترح:** ngrok / frp / Pinggy، Redis Streams أو AWS SQS، PostgreSQL replica  
**التعقيد:** ⭐⭐ منخفض-متوسط  
**ملف التكلفة:** منخفض-متوسط (الطبقة المدفوعة من ngrok للاستخدام الإنتاجي)

**المزايا:**
- سريع جدًا في النمذجة والنشر — وكيل ngrok ملف ثنائي واحد بلا تهيئة معقدة، مما يجعله مثاليًا لإثباتات المفهوم أو الطرح التدريجي
- البدائل المستضافة ذاتيًا (frp، Pinggy) تلغي الاعتماد على طرف ثالث ويمكن تشغيلها بالكامل ضمن البنية التحتية للمؤسسة
- مخزن قائمة الرسائل المؤقت يُضيف مرونة: إذا توقفت النسخة المتماثلة السحابية مؤقتًا، تُحتفَظ بالحمولات وتُعاد معالجتها دون فقدان بيانات
- نموذج مُشغِّل مرن — يمكن إرسال إشارة السحب بجدول أو عند الطلب أو استجابةً لأحداث التطبيق، مستوفيًا كلا مسارَي TR-2
- سهل المراجعة: يمكن تسجيل كل طلب سحب واستجابة على طبقة القائمة مع بيانات تعريف الحمولة الكاملة

**العيوب:**
- خطط ngrok للإنتاج (نطاقات مخصصة، أنفاق محجوزة، SSO) تحمل تكاليف شهرية ملموسة لكل نفق يمكن أن تنمو مع الحجم
- حجة امتثال TR-1 دقيقة — رغم أن جلسة TCP للخارج، فإن إشارة السحب تنبع من السحابة، مما قد يستلزم توضيحات إضافية خلال مراجعات الأمان
- بدائل النفق المستضافة ذاتيًا (frp) تتطلب من الفريق تشغيل وتأمين بنيتها التحتية الخاصة، مما يُضيف عبئًا تشغيليًا
- يُدخل الـ Relay خطوة شبكية إضافية وزمن تأخير محتمل مقارنةً بنفق مباشر للخارج (الحل الأول)
- أنفاق الطبقة المجانية في ngrok تستخدم عناوين URL عامة عشوائية تتغير عند إعادة التشغيل، مما يجعلها غير مناسبة للإنتاج بدون خطة النطاق المحجوز المدفوع

### الحل الرابع: Azure Arc مع خدمات البيانات + Azure Relay

**ملخص الهندسة المعمارية:**  
للمؤسسات المستثمرة بالفعل في منظومة Microsoft/Oracle، يمكن لـ **Azure Arc** إسقاط خدمات البيانات المحلية في المستوى الإداري لـ Azure دون الحاجة إلى اتصال وارد. يتيح **Azure Relay Hybrid Connection** لوكيل مزامنة OPERA المحلي التسجيل كمستمع. ترسل خدمة SaaS السحابية رسائل relay، يعالجها الوكيل محليًا (SELECT + إخفاء PII)، وينشر النتائج إلى **Azure SQL Read-Replica** أو **Azure Cosmos DB**. تُدار الحوكمة مركزيًا عبر Azure Policy وDefender for Cloud.

```
[OPERA Oracle DB]
        │ SELECT (read-only)
        ▼
[On-Prem Arc Agent + Relay Listener]
        │ Outbound registration to Azure Relay
        ▼
[Azure Relay Hybrid Connection Namespace]
        │
        ▼
[Azure Service Bus → Azure SQL / Cosmos DB]
        │
        ▼
[SaaS App Layer + Azure AD MFA (Entra ID)]
```

**خريطة الامتثال:**
- ✅ TR-1: مستمع Relay مُسجَّل للخارج؛ لا اتصالات واردة إلى الشبكة المحلية
- ✅ TR-2: مجدوَل أو مُشغَّل بالأحداث عبر رسائل Service Bus
- ✅ TR-3: Azure Relay لا يتطلب IP ثابت أو قواعد منافذ واردة
- ✅ SC-1: Azure Relay يفرض TLS 1.3 بشكل أصلي
- ✅ SC-2: إخفاء PII داخل وكيل Arc قبل الـ Relay
- ✅ SC-3: مستخدم Oracle مقيَّد بـ SELECT
- ✅ SA-1: Azure Entra ID يفرض MFA؛ سياسات الوصول الشرطي متاحة

**المكدس التقني المقترح:** Azure Arc، Azure Relay، Azure Service Bus، Azure SQL، Entra ID  
**التعقيد:** ⭐⭐⭐⭐ متوسط-مرتفع  
**ملف التكلفة:** متوسط-مرتفع (ترخيص Azure Arc + Relay)

**المزايا:**
- حوكمة مؤسسية من الدرجة الأولى: Azure Policy وDefender for Cloud وEntra ID Conditional Access توفر ضوابط أمان مركزية عبر جميع عقارات الفنادق المتصلة
- Entra ID MFA (SA-1) مُدمج بعمق، مما يتيح المصادقة بدون كلمة مرور وسياسات الوصول الشرطي وإدارة الهوية المميزة لحسابات المسؤولين
- Azure Arc يتيح مراقبة وتصحيح وإدارة الوكيل المحلي من بوابة Azure، مما يقلل العبء التشغيلي على فريق تقنية المعلومات في الفندق
- Azure Service Bus يوفر متانة رسائل مؤسسية وضمانات ترتيب وقوائم رسائل ميتة مدمجة — أكثر قوةً بكثير من قائمة مبسطة
- مناسب جدًا لمجموعات الفنادق المعيارية بالفعل على Microsoft 365 / Azure، متجنبًا علاقات بائع جديدة

**العيوب:**
- أعلى تعقيد إعداد بين حلول القراءة الخمسة؛ تهيئة Azure Arc وتوفير مساحة اسم Relay يتطلبان موظفين معتمدين من Azure
- ارتباط قوي ببائع Azure — لا يمكن تكرار الهندسة على AWS أو GCP دون إعادة بناء كاملة
- ترخيص Azure Arc وتكاليف Azure Relay Hybrid Connection تُضيف عبئًا شهريًا ثابتًا بصرف النظر عن حجم البيانات، مما يجعلها أقل كفاءةً من حيث التكلفة للنشرات أحادية العقار
- قد يتطلب تعيين مخطط Oracle إلى Azure SQL عملًا غير تافه في ترجمة أنواع البيانات، لا سيما لهياكل البيانات الخاصة بـ OPERA
- توثيق وأدوات Azure Relay المجتمعية أقل مقارنةً بـ Cloudflare Tunnel أو ngrok، مما يُبطئ استكشاف الأخطاء للفرق غير الخبيرة بـ Azure

### الحل الخامس: محرك سير عمل Temporal.io المستضاف ذاتيًا + شبكة WireGuard

**ملخص الهندسة المعمارية:**  
يُنسِّق محرك سير عمل **Temporal.io** المنشور في السحابة سير عمل مزامنة بيانات متينة وقابلة لإعادة المحاولة. تعمل عملية **Worker** من Temporal محليًا، مستطلِعةً خادم Temporal السحابي (للخارج فقط) بحثًا عن مهام سير العمل. عند تعيين مهمة، يستعلم Worker من قاعدة بيانات OPERA Oracle، ويُطبق إخفاء PII، ويُعيد النتيجة إلى خادم Temporal الذي يُثبِّتها في النسخة المتماثلة السحابية. يُقوَّى النقل الشبكي بـ **WireGuard** — VPN حديث متفوق تشفيريًا — لإنشاء شبكة ذات ثقة صفرية دون تعريض منافذ الشبكة المحلية.

```
[Temporal Cloud Server] ◄──── long-poll ────[On-Prem Temporal Worker]
                                                       │ (outbound only)
                                                       │ SELECT
                                                       ▼
                                              [OPERA Oracle DB]
                                                       │
                                            [PII Mask → Task Result]
                                                       │ WireGuard + TLS 1.3
                                                       ▼
                                           [Temporal Cloud Server]
                                                       │
                                                       ▼
                                         [Cloud Read-Replica + SaaS App]
```

**خريطة الامتثال:**
- ✅ TR-1: Worker يستطلع خادم Temporal للخارج؛ لا جلسات واردة من السحابة
- ✅ TR-2: جداول سير العمل قابلة للتهيئة؛ تدعم المُشغِّلات بالأحداث عبر Temporal Signals
- ✅ TR-3: WireGuard + استطلاع Worker يلغيان جميع متطلبات جدار الحماية الواردة
- ✅ SC-1: WireGuard (ChaCha20) + TLS 1.3 على نقل Temporal gRPC
- ✅ SC-2: إخفاء PII داخل نشاط Worker قبل إعادة نتيجة المهمة
- ✅ SC-3: بيانات اعتماد Oracle للقراءة فقط مُفعَّلة على مستوى قاعدة البيانات

**المكدس التقني المقترح:** Temporal Cloud، WireGuard، Python/Go Temporal SDK، Oracle cx_Oracle  
**التعقيد:** ⭐⭐⭐⭐ متوسط-مرتفع  
**ملف التكلفة:** متوسط (أسعار Temporal Cloud لكل إجراء)

**المزايا:**
- نموذج التنفيذ المتين في Temporal يضمن اكتمال كل سير عمل مزامنة في نهاية المطاف — تُعاد محاولة الخطوات الفاشلة تلقائيًا بتراجع قابل للتهيئة، مما يلغي فجوات البيانات الناتجة عن أعطال الشبكة العابرة
- WireGuard يوفر نقلًا حديثًا متفوقًا تشفيريًا (ChaCha20-Poly1305) بسطح هجوم ضئيل، مما يُقوي طبقة الشبكة بشكل كبير فوق TLS 1.3 وحده
- Temporal Signals تتيح مُشغِّلات حقيقية بالأحداث من أحداث أعمال OPERA دون الحاجة إلى تهيئة OXI، مستوفيةً المسار غير المتزامن في TR-2 عبر آلية مختلفة
- سجل سير العمل الكامل وإمكانية الإعادة في Temporal يوفران مسارًا تدقيقًا مدمجًا لكل عملية مزامنة، وهو قيّم لتقارير الامتثال
- كود سير العمل يمكن إدارة إصداراته واختباره ككود تطبيق، مما يجعله أكثر قابلية للصيانة على المدى البعيد مقارنةً بالمقاربات المعتمدة على cron أو قوائم الرسائل

**العيوب:**
- Temporal تقنية نادرة نسبيًا في قطاع الضيافة؛ توظيف أو التعاقد مع مطورين ذوي خبرة في Temporal أصعب من الأدوات السائدة
- تهيئة شبكة WireGuard تُضيف تعقيدًا تشغيليًا — إدارة المفاتيح وتدوير الأقران وتغييرات طبولوجيا الشبكة يجب إدارتها بعناية عبر مواقع فنادق متعددة
- أسعار Temporal Cloud قائمة على الاستهلاك (لكل إجراء في سير العمل)، مما يصعب التنبؤ به لبيئات OPERA عالية الحجم وقد يستلزم حدودًا للإنفاق
- مزيج Temporal + WireGuard + Oracle SDK يمثل أكبر عدد من التقنيات المتمايزة للتكامل والدعم في هذه المجموعة من الحلول
- Temporal المستضاف ذاتيًا (كبديل لـ Temporal Cloud) يتطلب تشغيل وصيانة مجموعة Cassandra أو PostgreSQL خلفية، مما يزيد البنية التحتية بشكل كبير

## 3. حلول القراءة والكتابة <a name="read-write-solutions"></a>

> ⚠️ **ملاحظة امتثال حرجة:** يُلزم PRD (SC-3) بتقييد مستخدم تكامل Oracle بصلاحيات `SELECT`. **الكتابة العكسية إلى قاعدة بيانات Oracle الإنتاجية لـ OPERA خارج النطاق صراحةً.** الحلول التالية تعالج الوصول للكتابة إلى **طبقة البيانات السحابية المُدارة بواسطة SaaS** (مثل النسخة المتماثلة، جداول الإثراء، حالة سير العمل)، وليس إلى قاعدة بيانات OPERA نفسها. حيث تكون الكتابة العكسية إلى OPERA متطلبًا حقيقيًا للأعمال، يجب توفير حساب خدمة مميز منفصل مع صلاحيات `INSERT/UPDATE` مُدققة ومحددة النطاق وسير عمل موافقة على التغييرات — وهذا يُعد **تعديلًا على PRD** يستلزم التوقيع عليه.

### حل القراءة/الكتابة (أ): نفق Cloudflare ثنائي الاتجاه + نمط قائمة الأوامر
*(يمتد على الحل الأول)*

**ملخص الهندسة المعمارية:**  
يمتد هذا الحل على هندسة نفق Cloudflare بنمط **قائمة أوامر غير متزامنة**. يكتب أصحاب المصلحة الخارجيون أو تطبيق SaaS أوامر (مثل تجاوزات الأسعار والملاحظات والعلامات) إلى **قائمة أوامر** جانب السحابة (AWS SQS أو Azure Service Bus). يستطلع الوكيل المحلي — الذي يحافظ بالفعل على النفق للخارج — هذه القائمة دوريًا للأوامر المعلقة، ويتحقق من صحتها مقابل مخطط قائمة مسموح بها، ويُطبق الكتابة على **جداول OPERA المعتمدة غير الإنتاجية** (مثل مخطط التدريج أو قاعدة بيانات sidecar مُدارة بواسطة SaaS).

```
[SaaS User / Stakeholder]
        │ Write intent (MFA-authenticated)
        ▼
[Cloud Command Queue (SQS)]
        │
        ▼                         ◄── Outbound poll (same Cloudflare Tunnel)
[On-Prem Agent]
        │ Schema validation + allowlist check
        │
        ├─► [OPERA Oracle DB — SELECT only, no change]
        │
        └─► [SaaS Sidecar DB / Approved Staging Schema — INSERT/UPDATE permitted]
```

**نطاق الكتابة:** جداول sidecar المُدارة بواسطة SaaS فقط. يظل مخطط OPERA الأساسي للقراءة فقط.  
**ضوابط إضافية مطلوبة:**
- التحقق من مخطط الأوامر (رفض الحمولات المشوهة/غير المتوقعة)
- سجل تدقيق لجميع عمليات الكتابة مع نسب المستخدم
- تحديد معدل على قائمة الأوامر لمنع الإساءة
- حساب خدمة مميز منفصل لأي مسار كتابة (مختلف عن حساب القراءة)

**خريطة الامتثال:**
- ✅ TR-1: الوكيل يستطلع القائمة للخارج؛ لا اتصال سحابي وارد إلى الشبكة المحلية
- ✅ SC-3: مستخدم Oracle في OPERA يظل للقراءة فقط؛ الكتابة تذهب إلى مخطط sidecar
- ✅ SA-1: نية الكتابة مُصادَق عليها بـ MFA على طبقة SaaS

**المزايا:**
- يُعيد استخدام البنية التحتية الحالية لنفق Cloudflare من الحل الأول، مما يُقلل المكونات الإضافية وسطح التشغيل
- قائمة الأوامر غير المتزامنة تفصل بين نية الكتابة والتنفيذ — يمكن لأصحاب المصلحة تقديم أوامر حتى عندما يكون الوكيل المحلي متوقفًا مؤقتًا، مع ضمان التسليم عند إعادة الاتصال
- التحقق من المخطط وفرض القائمة المسموح بها على مستوى الوكيل يوفر طبقة دفاعية قوية قبل وصول أي بيانات إلى الأنظمة المحلية
- حسابات خدمة منفصلة لمسارَي القراءة والكتابة (كما هو مطلوب) تحدد نصف قطر الانفجار عند اختراق بيانات الاعتماد
- نمط معروف جيدًا (معالجة الأوامر القائمة على قوائم الرسائل) بدعم واسع من الأدوات عبر AWS SQS وAzure Service Bus والبدائل مفتوحة المصدر

**العيوب:**
- نطاق الكتابة محدود عمدًا بمخطط sidecar — إذا احتاج أصحاب المصلحة إلى الكتابة في جداول OPERA الأصلية، فهذا الحل غير كافٍ دون تعديل PRD وبيانات اعتماد Oracle إضافية
- يُدخل عملية ثانية محلية (معالج الأوامر) يجب مراقبتها وصيانتها ومزامنتها مع وكيل القراءة
- إدارة مخطط القائمة المسموح بها مسؤولية مستمرة؛ مع تطور تطبيق SaaS، يجب تحديث القائمة وإعادة نشرها للوكيل المحلي
- التنفيذ غير المتزامن للكتابة يعني عدم تلقي أصحاب المصلحة تأكيدًا فوريًا بتطبيق أوامرهم — يجب أن يراعي تصميم واجهة المستخدم أنماط التغذية الراجعة للاتساق النهائي
- تحديد المعدل والوقاية من الإساءة على قائمة الأوامر يجب ضبطهما بعناية؛ مقيّد جدًا تتأخر الكتابات المشروعة، مُتسامح جدًا تصبح القائمة ناقلًا محتملًا لهجمات الحرمان من الخدمة

### حل القراءة/الكتابة (ب): بوابة OHIP REST API مع الكتابة العكسية المحددة النطاق
*(يمتد على الحل الثاني)*

**ملخص الهندسة المعمارية:**  
توفر **OHIP (منصة تكامل Oracle للضيافة)** من Oracle طبقة REST API معتمدة فوق OPERA Cloud. حيث انتقل الفندق إلى OPERA Cloud أو تكاملًا جزئيًا معه، يمكن لـ OHIP أن يكون **قناة الكتابة الرسمية المعتمدة** إلى OPERA — مستبدلًا الكتابة المباشرة في قاعدة Oracle بطفرات على مستوى API يتحقق منها OPERA نفسه. **بوابة API** (Kong أو AWS API Gateway) تُقدَّم أمام OHIP، مفرضةً نطاقات OAuth 2.0 وحدود المعدل وسجلات التدقيق. يحمل تطبيق SaaS رموزًا محددة النطاق تسمح فقط بنقاط نهاية OHIP المعتمدة (مثل `PUT /reservations/{id}/notes`).

```
[SaaS App — MFA Authenticated User]
        │ Scoped OAuth 2.0 token
        ▼
[Cloud API Gateway (Kong / AWS APIGW)]
        │ Scope enforcement + rate limit + audit
        ▼
[OHIP REST API (Oracle-managed)]
        │ Validated write to OPERA
        ▼
[OPERA PMS / Oracle DB]
```

**نطاق الكتابة:** العمليات المسموح بها في OHIP فقط (OPERA يتحقق من جميع الطفرات).  
**ضوابط إضافية مطلوبة:**
- تصنيف نطاق OAuth مُعيَّن لصلاحيات نقاط نهاية OHIP
- مسار تدقيق كامل على مستوى بوابة API
- اشتراك/ترخيص OHIP من Oracle مطلوب
- التأكد من أن إصدار OPERA في الفندق يدعم تكامل OHIP

**خريطة الامتثال:**
- ✅ TR-1: القراءات تظل قائمة على السحب؛ الكتابات عبر OHIP قناة منفصلة معتمدة
- ✅ SC-3: لا كتابات SQL مباشرة في Oracle؛ جميع الطفرات تمر عبر طبقة تحقق OHIP
- ✅ SA-1: OAuth 2.0 + MFA على طبقة SaaS؛ OHIP يوفر مصادقة إضافية

**المزايا:**
- الحل الوحيد الذي يكتب في هياكل بيانات OPERA الأصلية عبر قناة معتمدة رسميًا من Oracle — مما يلغي خطر إتلاف قاعدة البيانات الإنتاجية بـ SQL مشوه
- طبقة التحقق الخاصة بـ OHIP تعمل كخط دفاع ثانٍ، رافضةً الكتابات التي تنتهك قواعد أعمال OPERA (مثل حالات الحجز غير الصالحة أو رموز الأسعار المكررة)
- نطاقات OAuth 2.0 توفر تحكمًا دقيقًا وقابلًا للتدقيق — يمكن تقييد كل دور لصاحب مصلحة بنقاط نهاية OHIP التي تتطلبها وظيفته بالضبط
- تحديد المعدل وسجلات التدقيق على مستوى بوابة API تُمركز ضوابط الأمان دون الحاجة إلى تغييرات محلية
- أفضل توافق على المدى البعيد مع خارطة طريق منتجات Oracle، إذ تُعد OHIP معيار التكامل المُعلَن من Oracle لـ OPERA Cloud مستقبلًا

**العيوب:**
- يتطلب اشتراكًا نشطًا في OHIP وترخيص Oracle، مما يُضيف تكلفة بائع كبيرة ومستمرة — ومفاوضات ترخيص Oracle المؤسسية قد تكون مطوَّلة
- يعتمد كليًا على نشر الفندق لـ OPERA Cloud أو تهيئة هجينة متوافقة مع OHIP؛ تثبيتات OPERA المحلية فقط قد لا تدعم OHIP دون ترقية
- نقاط نهاية الكتابة المتاحة في OHIP محدودة بما نشرته Oracle — قد لا تكون عمليات OPERA المخصصة أو الفريدة قابلةً للتحقيق عبر هذه القناة
- إدارة بوابة API (Kong أو AWS APIGW) مكوّن بنية تحتية إضافي يتطلب نشره ومقياسه وانضباط صيانته الخاص
- تغييرات إصدار OHIP API قد تكسر التكاملات إذا لم يتتبع تطبيق SaaS دورة إصدار Oracle، مما يخلق اعتمادًا على جدول تحديثات Oracle

### حل القراءة/الكتابة (ج): هندسة CQRS القائمة على مصدر الأحداث مع وكيل ذي مسار مزدوج
*(يمتد على الحلَّيْن الأول والخامس)*

**ملخص الهندسة المعمارية:**  
هذا هو الخيار الأكثر نضجًا معماريًا. يفصل نمط **CQRS (فصل مسؤولية الأوامر والاستعلامات)** اهتمامات القراءة والكتابة كليًا. **مسار الاستعلام** مطابق للحل الأول أو الخامس — يسحب الوكيل المحلي بيانات OPERA ويُعبئ النسخة المتماثلة السحابية. **مسار الأوامر** يحتفظ بـ **مخزن أحداث** جانب السحابة (مثل EventStoreDB أو Kafka) حيث تُسجَّل نوايا الكتابة كأحداث غير قابلة للتغيير. **معالج أوامر** منفصل محليًا يشترك في هذا التدفق (اتصال للخارج)، ويُطبق التحقق من قواعد الأعمال، وينفذ الكتابات المعتمدة مقابل **مخطط مُتحكَّم به بواسطة SaaS** داخل بيئة الفندق أو عبر OHIP.

```
              ┌─────── QUERY PATH (Read) ──────────────────────────┐
              │                                                     │
[OPERA Oracle DB] ──SELECT──► [On-Prem Read Agent] ──► [Cloud Replica]
                                                                    │
                                                         [SaaS Read UI]

              ┌─────── COMMAND PATH (Write) ────────────────────────┐
              │                                                     │
[SaaS Write UI] ──► [Cloud Event Store / Kafka] ◄── poll (outbound)
                                                         │
                                              [On-Prem Command Processor]
                                                         │ Validates + routes
                                                         ├─► [Sidecar Schema / OHIP]
                                                         └─► [Audit Log]
```

**نطاق الكتابة:** مخطط sidecar لـ SaaS و/أو نقاط نهاية OHIP API. مخزن الأحداث يوفر قابلية تدقيق كاملة وإمكانية إعادة التشغيل.  
**ضوابط إضافية مطلوبة:**
- سجل أحداث غير قابل للتغيير (إضافة فقط) لمسار تدقيق كامل
- قائمة رسائل ميتة للأوامر الفاشلة/المرفوضة
- هوية عملية محلية منفصلة لمعالج الأوامر مقابل وكيل القراءة
- استراتيجية تحديد إصدار المخطط لحمولات الأحداث

**خريطة الامتثال:**
- ✅ TR-1: كلا معالجَي القراءة والأوامر يستخدمان الاستطلاع للخارج؛ لا اتصالات واردة إلى الشبكة المحلية
- ✅ TR-3: كلا الوكيلَيْن يتصلان للخارج عبر نفق/relay — لا IP ثابت مطلوب
- ✅ SC-1: Kafka/EventStore + TLS 1.3 على جميع مسارات النقل
- ✅ SC-3: مستخدم OPERA Oracle DB يظل للقراءة فقط؛ لا كتابات SQL مباشرة من السحابة
- ✅ SA-1/SA-2: النسخة المتماثلة للقراءة ومخزن الأحداث منفصلان؛ أحداث الكتابة مُصادَق عليها بـ MFA + رموز محددة النطاق

**المزايا:**
- الهندسة الأكثر قابلية للتدقيق في هذا المستند: كل نية كتابة تُسجَّل كحدث غير قابل للتغيير قبل التنفيذ، مما يتيح إعادة التشغيل الكاملة والتحقيق الجنائي وإعداد التقارير التنظيمية
- الفصل الصارم في CQRS بين مسارَي القراءة والكتابة يلغي خطر تدهور أداء القراءة على خادم OPERA أو النسخة المتماثلة السحابية جراء عملية كتابة
- مخزن الأحداث يتيح الاستعلام الزمني — يمكن لأصحاب المصلحة إعادة بناء حالة أي كيان في OPERA في أي نقطة زمنية، وهو قيّم لتسوية النزاعات وتقارير المساهمين
- معالجة قائمة الرسائل الميتة وإعادة تشغيل الأحداث تجعل النظام مرنًا للغاية؛ يمكن فحص الأوامر الفاشلة وتصحيحها وإعادة معالجتها دون فقدان بيانات
- يتوسع بشكل طبيعي لنشرات متعددة العقارات — يمكن إعداد مواقع فنادق إضافية كمجموعات مستهلكين جديدة على نفس موضوع Kafka دون تغييرات معمارية

**العيوب:**
- أعلى تعقيد هندسي في هذا المستند؛ CQRS ومصادر الأحداث أنماط متقدمة تتطلب مطورين ذوي خبرة محددة في الأنظمة الموزعة للتنفيذ الصحيح
- العبء التشغيلي جوهري: Kafka (أو EventStoreDB) يتطلب نشره ومراقبته وإدارة سياسة الاحتفاظ وتخطيط الطاقة الخاصة به
- تحديد إصدار مخطط حمولات الأحداث التزام صيانة طويل الأمد — مع تطور نموذج بيانات OPERA أو تطبيق SaaS، يجب تحديد إصدار مخططات الأحداث والحفاظ على توافق المستهلكين القدامى
- عمليتان منفصلتان للوكيل المحلي (وكيل القراءة + معالج الأوامر) تضاعفان بصمة النشر والمراقبة المحلية
- الاتساق النهائي بين النسخة المتماثلة للقراءة ونتائج الأوامر قد يكون محيِّرًا للمستخدمين النهائيين إذا لم تُوضِّح واجهة المستخدم الحالات المعلقة مقابل المطبقة بوضوح — يتطلب تصميم UX دقيقًا

## 5. نقطة البداية الموصى بها

بالنسبة لمعظم نشرات الفنادق، يُقدم **الحل الأول (Cloudflare Tunnel + Replica)** أفضل توازن بين البساطة والتكلفة وتغطية الامتثال. إذا كانت أحداث الأعمال مُهيَّأة بالفعل في OXI/OHIP، يوفر **الحل الثاني** نضارة بيانات متفوقة دون تكبد عبء الاستطلاع.

بالنسبة للمؤسسات التي تتطلب الكتابة العكسية، يُقدم **الحل (أ)** (توسيع الحل الأول بقائمة أوامر) المسار الأدنى مخاطرةً، إذ يحافظ على قيد القراءة فقط على قاعدة بيانات OPERA Oracle مع تمكين الكتابة المتحكم بها إلى sidecar مُدار بواسطة SaaS.

يُوصى بـ **الحل (ج) CQRS** فقط حيث تُبرر قابلية التدقيق طويلة الأمد أو النطاق متعدد العقارات أو سير عمل الكتابة المعقدة الاستثمار الهندسي الإضافي.
