# التكامل الهجين الآمن لنظام SaaS مع OPERA PMS
## خيارات هندسة الحلول — نسخة مُحدَّثة

**إصدار الوثيقة:** 2.0  
**الحالة:** مسودة  
**إصدار وثيقة متطلبات المنتج المرجعية:** 1.0 (مُعدَّل)  
**يحل محل:** الإصدار 1.0

---

## سجل التغييرات من الإصدار 1.0 إلى 2.0

| القيد | الإصدار 1.0 | الإصدار 2.0 |
|---|---|---|
| **TR-1** | المزامنة عبر السحب فقط؛ لا دفع من السحابة | ~~محذوف~~ — المزامنة ثنائية الاتجاه مسموح بها الآن |
| **TR-2** | فترة المزامنة ≥ ساعة أو أحداث OXI/OHIP غير متزامنة | **مُستبدَل** — يجب أن يبقى OPERA في مزامنة فورية مستمرة مع خدمة MERN السحابية |
| **TR-3** | لا يُسمح بعنوان IP ثابت أو إعادة توجيه المنافذ | **مُخفَّف** — عنوان IP الثابت مسموح به الآن حيث يكون مبررًا تشغيليًا |
| **SC-3** | مستخدم قاعدة البيانات مقيَّد بـ `SELECT` فقط | **مُوسَّع** — `INSERT` و`UPDATE` مطلوبان الآن للحجوزات على أقل تقدير |
| **نطاق الكتابة** | SaaS sidecar / OHIP فقط | **مُوسَّع** — الكتابة في جداول حجوزات OPERA متطلب أساسي |

## 1. ملخص القيود المُحدَّثة <a name="overview"></a>

| القيد | المتطلب |
|---|---|
| **TR-1** | المزامنة ثنائية الاتجاه مسموح بها؛ أنماط الدفع والسحب كلاهما مقبولان |
| **TR-2** | يجب أن يبقى OPERA في **مزامنة فورية ومستمرة** مع خدمة MERN السحابية — الاتساق النهائي غير مقبول للحجوزات والخدمات |
| **TR-3** | عنوان IP الثابت وإعادة توجيه المنافذ مسموح بهما في موقع الفندق حيث يكون ذلك مبررًا تشغيليًا |
| **SC-1** | TLS 1.3 لجميع البيانات أثناء النقل — غير قابل للتفاوض |
| **SC-2** | إخفاء أو تجزئة بيانات التعريف الشخصية (PII) قبل مغادرتها البيئة المحلية ما لم تكن مطلوبة لوظيفة الحجز الأساسية |
| **SC-3** | يجب أن يدعم مستخدم قاعدة البيانات التكاملية `SELECT` و`INSERT` و`UPDATE` على جداول الحجوزات على أقل تقدير؛ `DELETE` يظل محظورًا |
| **SA-1** | إلزامية المصادقة متعددة العوامل (MFA) لجميع المستخدمين الخارجيين |
| **SA-2** | قراءات أصحاب المصلحة العامة تستهدف النسخة المتماثلة السحابية؛ عمليات كتابة الحجوزات تستخدم مسار كتابة مخصصًا |

> **ملاحظة مهمة بشأن SC-2 والحجوزات:** اسم الضيف وبيانات الاتصال ومعلومات الهوية مطلوبة لعمليات الحجز. هذه الحقول مُستثناة من إخفاء PII على مسار الكتابة، لكن يجب تشفيرها أثناء النقل (SC-1) وفي حالة السكون في قاعدة البيانات السحابية.

> **ملاحظة مهمة بشأن SC-3:** يجب توفير مستخدم قاعدة بيانات مخصص بصلاحيات `INSERT`/`UPDATE` محددة النطاق على جداول `RESERVATION` و`RESERVATION_NAME` و`ALLOTMENT` والجداول ذات الصلة، بشكل منفصل عن مستخدم التقارير للقراءة فقط. تظل عمليات `DELETE` و`DROP` و`TRUNCATE` وأوامر DDL محظورة صراحةً.

## 2. الحلول الهجينة للقراءة والكتابة <a name="solutions"></a>

---

### الحل الأول: OHIP REST API (ثنائي الاتجاه) + الواجهة الخلفية MERN + مزامنة MongoDB Atlas

**ملخص الهندسة المعمارية:**
الخيار الأكثر توافقًا مع Oracle. تتواصل خدمة MERN السحابية مع OPERA حصريًا عبر **OHIP REST API من Oracle** — طبقة التكامل المعتمدة رسميًا لـ OPERA Cloud. تستدعي الواجهة الخلفية Node.js نقاط نهاية OHIP لقراءة بيانات التوافر والحجوزات، ولكتابة حجوزات جديدة مباشرةً في OPERA. يكتشف **MongoDB Change Stream** على Atlas طفرات الحجوزات ويُشغِّل استدعاءات كتابة OHIP الصادرة في شبه وقت فعلي، مما يُبقي النظامين في مزامنة مستمرة وفق TR-2. يُستخدم عنوان IP ثابت في شبكة الفندق لإدراج حركة OHIP الواردة في القائمة البيضاء، مستوفيًا TR-3 المُخفَّف.

```
[MERN React Frontend]
        │ REST
        ▼
[Node.js Express Backend]
        │ OHIP OAuth 2.0 token
        ├──────────────────────────────────────────────┐
        │ Read: GET /reservations, /availability        │ Write: POST /reservations
        ▼                                              ▼
[OHIP REST API — Oracle Managed]──────────────────────►[OHIP REST API]
        │                                              │
        ▼                                              ▼
[OPERA Oracle DB — Read via OHIP]          [OPERA Oracle DB — Write via OHIP]
        │
        ▼
[MongoDB Atlas — Read Replica / Cache]
        │ Change Streams
        ▼
[Stakeholder Dashboard (SA-2)]
```

**آلية المزامنة:** يعمل MongoDB Atlas كذاكرة تخزين مؤقت للقراءة تُغذَّى بالاستطلاع عبر OHIP أو الـ Webhooks. تتجاوز عمليات الكتابة MongoDB كليًا وتذهب مباشرةً OHIP ← OPERA لضمان الاتساق التعاملي. تُخطر Webhooks OHIP الواجهة الخلفية MERN بالتغييرات من جانب OPERA (مثل الوصول المباشر دون حجز والتعديلات من نظام PMS) لإبقاء Atlas محدَّثًا.

**خريطة الامتثال:**
- ✅ TR-1: ثنائي الاتجاه عبر OHIP API؛ القراءة والكتابة مدعومتان بشكل أصلي
- ✅ TR-2: Webhooks OHIP تُوصِّل تغييرات حالة OPERA إلى MERN في شبه وقت فعلي؛ تأكيدات الكتابة متزامنة
- ✅ TR-3: عنوان IP ثابت في القائمة البيضاء لتهيئة OHIP؛ لا إعادة توجيه منافذ خام لقاعدة Oracle
- ✅ SC-1: OHIP يفرض TLS 1.3؛ Atlas TLS 1.3 أثناء النقل
- ✅ SC-2: PII تُرسَل فقط لعمليات الحجز حيث يكون ذلك مطلوبًا
- ✅ SC-3: لا بيانات اعتماد Oracle مباشرة — جميع الوصول يتم عبر OHIP الذي يفرض نموذج صلاحياته الخاص
- ✅ SA-1/SA-2: MFA على طبقة مصادقة MERN؛ عروض أصحاب المصلحة تستخدم نسخة Atlas المتماثلة

**المكدس التقني:** OHIP REST API، Node.js/Express، MongoDB Atlas، React، Atlas Change Streams  
**التعقيد:** ⭐⭐⭐ متوسط  
**ملف التكلفة:** متوسط-مرتفع (اشتراك OHIP + مجموعة Atlas M10+)

**المزايا:**
- الكتابات إلى OPERA تُتحقق منها طبقة المنطق التجاري الخاصة بـ Oracle — سترفض OHIP حالات الحجز غير الصالحة والحجوزات المزدوجة وانتهاكات خطط الأسعار قبل وصولها إلى قاعدة البيانات
- لا بيانات اعتماد Oracle مباشرة مطلوبة في كود MERN؛ سطح الهجوم يتقلص بشكل جذري
- Webhooks OHIP توفر إشعارًا فوريًا حقيقيًا من OPERA إلى الواجهة الخلفية MERN، مستوفيًا متطلب المزامنة المستمرة في TR-2 دون عبء الاستطلاع
- أنظف قصة صيانة على المدى البعيد — مع تطور Oracle لـ OPERA Cloud، تمتص OHIP تغييرات المخطط وتُعزَل الواجهة الخلفية MERN
- Atlas Change Streams تمنح واجهة React طبقة بيانات فورية عبر WebSocket بحد أدنى من كود الواجهة الخلفية

**العيوب:**
- اعتماد صارم على اشتراك OHIP وترخيص Oracle — الأغلى في هذه المجموعة، وقد تستغرق عملية ترخيص Oracle المؤسسية أسابيع
- OHIP متاح فقط لنشرات OPERA Cloud؛ تثبيتات OPERA 5.x المحلية تتطلب نهج OPERA Web Services (OWS) منفصل
- نقاط نهاية الكتابة المنشورة في OHIP قد لا تغطي كل حالات الحجز الحافية — التخصيصات الجماعية المعقدة والباقات والغرف المتعددة قد تتطلب حلولًا بديلة
- الواجهة الخلفية MERN يجب أن تُنفِّذ تجديد رمز OHIP ومنطق إعادة المحاولة والتحقق من توقيع Webhook، مما يُضيف كودًا تمهيديًا تكاملًا غير تافه
- تعتمد المزامنة الفورية على موثوقية Webhook في OHIP؛ توقف Webhook يعني انجراف نسخة Atlas حتى دورة الاستطلاع التالية

### الحل الثاني: اتصال Oracle DB مباشر عبر نفق مشفر + طبقة ORM في MERN

**ملخص الهندسة المعمارية:**
تتصل الواجهة الخلفية Node.js في MERN **مباشرةً بقاعدة بيانات OPERA Oracle** عبر نفق مُقوَّى ومشفر — إما **VPN موقع إلى موقع عبر IPSec** بين شبكة الفندق ومزود السحابة، أو **اتصال TCP مُغلَّف بـ SSL** عبر عنوان IP الثابت للفندق. يُوفَّر مستخدم Oracle DB مخصص بصلاحيات `SELECT` و`INSERT` و`UPDATE` المحددة النطاق بإحكام على جداول الحجوزات فقط. تستخدم الواجهة الخلفية Node.js مكتبة عميل Oracle (مثل `node-oracledb`) لتنفيذ استعلامات مُعامَلة الحجج، مع طبقة ORM رقيقة تفرض مجموعة العمليات المسموح بها. يخدم MongoDB Atlas كنسخة متماثلة للقراءة تُغذَّى بعملية التقاط تغييرات البيانات (CDC) من جانب Oracle.

```
[MERN React Frontend]
        │
        ▼
[Node.js Express Backend]
        │ node-oracledb — TLS 1.3 over IPSec VPN / Static IP
        ▼
[Hotel Network — Static IP Endpoint]
        │
        ▼
[OPERA Oracle DB]
   ├── READ: SELECT on reservations, availability, rates
   └── WRITE: INSERT/UPDATE on RESERVATION, RESERVATION_NAME tables
        │
        ▼ (CDC / LogMiner / GoldenGate)
[MongoDB Atlas Read Replica]
        │
        ▼
[Stakeholder Dashboard]
```

**آلية المزامنة:** Oracle LogMiner أو GoldenGate CDC يبثان أحداث التغيير من OPERA Oracle DB إلى MongoDB Atlas في شبه وقت فعلي. تذهب الكتابات من الواجهة الخلفية MERN مباشرةً إلى Oracle وتنعكس فورًا في القراءات اللاحقة، محققًا المزامنة المستمرة المطلوبة في TR-2 دون تأخير الاستطلاع.

**خريطة الامتثال:**
- ✅ TR-1: اتصال DB مباشر ثنائي الاتجاه؛ السحابة تبدأ كلًا من القراءات والكتابات
- ✅ TR-2: CDC يوفر نشرًا دون الثانية لتغييرات OPERA إلى Atlas؛ الكتابات المباشرة فورية الاتساق
- ✅ TR-3: عنوان IP ثابت مستخدم لنقطة نهاية VPN أو إدراج مستمع Oracle في القائمة البيضاء
- ✅ SC-1: IPSec VPN + تشفير Oracle الأصلي (TLS 1.3 / AES-256) على جميع اتصالات قاعدة البيانات
- ✅ SC-2: PII مُتحكَّم بها على طبقة التطبيق؛ حقول الحجز تُرسَل حسب الحاجة
- ✅ SC-3: مستخدم Oracle مخصص بـ `SELECT` و`INSERT` و`UPDATE` على الجداول المحددة النطاق؛ لا `DELETE` أو DDL
- ✅ SA-1/SA-2: MFA على طبقة MERN؛ قراءات أصحاب المصلحة تُخدَم من نسخة Atlas المتماثلة

**المكدس التقني:** node-oracledb، Oracle LogMiner / GoldenGate، MongoDB Atlas، IPSec VPN، React  
**التعقيد:** ⭐⭐⭐⭐ متوسط-مرتفع  
**ملف التكلفة:** متوسط (بنية تحتية VPN + ترخيص GoldenGate إن استُخدم)

**المزايا:**
- الوصول المباشر لقاعدة البيانات يمنح الواجهة الخلفية MERN تحكمًا كاملًا في تصميم الاستعلامات وإدارة المعاملات وعمليات الكتابة — لا قيود طبقة API تحدّ مما يمكن التعبير عنه
- CDC القائم على LogMiner هو آلية المزامنة المستمرة الأكثر موثوقية المتاحة لـ Oracle — تُلتقط التغييرات على مستوى سجل الإعادة، مما يعني عدم فقدان أي شيء حتى لو توقفت طبقة التطبيق مؤقتًا
- أدنى زمن تأخير للقراءة من أي حل في هذا المستند — الواجهة الخلفية MERN يمكنها الاستعلام من Oracle مباشرةً بوقت رحلة ذهاب وإياب دون الميلي ثانية على VPN
- عنوان IP الثابت + IPSec VPN ضابط أمان شبكي معروف وقابل للتدقيق يألفه فريقا تقنية المعلومات وOracle DBA
- لا اعتماد على SaaS طرف ثالث في مسار البيانات — التكامل داخل تحكم البنية التحتية الخاصة بك بالكامل

**العيوب:**
- بيانات اعتماد Oracle المباشرة في البيئة السحابية تمثل أعلى خطر أمني في هذه المجموعة؛ واجهة MERN خلفية مخترقة لديها مسار كتابة مباشر إلى قاعدة بيانات OPERA الإنتاجية
- يتطلب Oracle DBA لتوفير مستخدم قاعدة البيانات المحدد النطاق وصيانته وإدارة حجم مجمع الاتصالات ومراقبة تغييرات مخطط OPERA التي قد تكسر الاستعلامات
- Oracle GoldenGate (خيار CDC الإنتاجي) منتج Oracle مُرخَّص بتكلفة كبيرة؛ LogMiner مجاني لكنه أكثر تعقيدًا تشغيليًا للصيانة على نطاق واسع
- تهيئة IPSec VPN بين شبكة الفندق ومزود السحابة تتطلب تنسيقًا بين فريق تقنية المعلومات ومزود السحابة وربما شركة إدارة العقارات
- أي ترقية لـ OPERA تُغيِّر مخطط جدول الحجوزات قد تكسر استعلامات أو كتابات MERN بصمت — يتطلب عملية مراقبة تغييرات المخطط

### الحل الثالث: وكيل Node.js المحلي + دفع WebSocket + MongoDB Atlas

**ملخص الهندسة المعمارية:**
يعمل **وكيل Node.js** على الجهاز المحلي داخل شبكة الفندق، متصلًا بكلٍّ من OPERA Oracle DB محليًا وخدمة MERN السحابية عبر **اتصال WebSocket أو Server-Sent Events (SSE) دائم**. نظرًا لأن الوكيل يبدأ اتصال WebSocket للخارج، لا يلزم إعادة توجيه المنافذ — وإن كان عنوان IP الثابت متاحًا إذا احتُيج إليه لاستقرار الاتصال. يستمع الوكيل لأحداث تغيير Oracle (عبر الاستطلاع أو مُشغِّلات OXI)، ويُسلسِل الحمولة، و**يدفع التحديثات إلى الواجهة الخلفية MERN في وقت فعلي**. لعمليات الكتابة، تُرسِل الواجهة الخلفية MERN أوامر إنشاء/تحديث الحجوزات عبر نفس قناة WebSocket، ويُطبِّقها الوكيل مباشرةً على Oracle DB باستخدام بيانات اعتماد محددة النطاق.

```
[OPERA Oracle DB]◄──INSERT/UPDATE──[On-Prem Node.js Agent]
        │                                    ▲│
        │ Poll / OXI trigger                  ││ WebSocket (persistent, outbound)
        ▼                                    │▼
[Change Detected]                   [MERN Node.js Backend]
        │                                    │
        └──── serialise + push ─────────────►│
                                             │
                                    [MongoDB Atlas]
                                             │
                                    [React Frontend]
                                             │
                                    [Stakeholder Dashboard]
```

**آلية المزامنة:** اتصال WebSocket كامل الازدواجية — تغييرات OPERA تتدفق للأعلى إلى الواجهة الخلفية MERN باستمرار (مستوفيًا TR-2)، وأوامر كتابة الحجوزات تتدفق للأسفل إلى الوكيل للتنفيذ الفوري محليًا. يُقرّ الوكيل الكتابات برسالة تأكيد، مانحًا الواجهة الخلفية MERN تغذية راجعة تعاملية.

**خريطة الامتثال:**
- ✅ TR-1: الوكيل يبدأ WebSocket للخارج؛ الواجهة الخلفية MERN يمكنها دفع الأوامر عبر القناة المُنشأة
- ✅ TR-2: دفع ثنائي الاتجاه فوري عبر WebSocket دائم؛ لا تأخير استطلاع لتغييرات الحالة
- ✅ TR-3: عنوان IP الثابت متاح لكن غير مطلوب؛ WebSocket صادر من الوكيل
- ✅ SC-1: WSS (WebSocket Secure) يفرض TLS 1.3؛ اتصال Oracle يستخدم التشفير الأصلي
- ✅ SC-2: PII مُدرَجة فقط في حمولات الحجوزات حيث يكون ذلك مطلوبًا
- ✅ SC-3: الوكيل يستخدم بيانات اعتماد Oracle محددة النطاق: `SELECT` للقراءات، `INSERT`/`UPDATE` على جداول الحجوزات للكتابات
- ✅ SA-1/SA-2: MFA على طبقة مصادقة MERN؛ Atlas يخدم قراءات أصحاب المصلحة

**المكدس التقني:** Node.js (وكيل محلي + واجهة MERN الخلفية)، مكتبة WebSocket/ws، Oracle node-oracledb، MongoDB Atlas، React  
**التعقيد:** ⭐⭐⭐ متوسط  
**ملف التكلفة:** منخفض-متوسط (لا ترخيص إضافي؛ تكلفة حوسبة لعملية الوكيل)

**المزايا:**
- الوكيل المحلي مكتوب بـ Node.js — نفس لغة الواجهة الخلفية MERN — مما يعني أن نفس فريق التطوير يمكنه امتلاك ونشر وصيانة طرفَي التكامل دون التبديل بين السياقات
- WebSocket كامل الازدواجية يوفر مزامنة ثنائية الاتجاه فورية حقيقية، مستوفيًا TR-2 دون تعقيد خطوط القراءة والكتابة المنفصلة
- لا بيانات اعتماد Oracle تغادر شبكة الفندق — الوكيل يحتفظ ببيانات اعتماد قاعدة البيانات محليًا؛ الواجهة الخلفية MERN تُرسِل حمولات الأوامر فقط، لا تلمس قاعدة البيانات مباشرةً
- عنوان IP الثابت متاح لكن الهندسة لا تعتمد عليه — إذا تغير IP الفندق، يُعيد الوكيل الاتصال للخارج تلقائيًا
- خفيف وقليل التكلفة — الوكيل عملية Node.js بلا متطلبات ترخيص إضافية سوى مكتبة عميل Oracle الموجودة

**العيوب:**
- يجب الحفاظ على اتصال WebSocket دائم بين الفندق والسحابة عبر انقطاعات الشبكة وعدم استقرار Wi-Fi الفندق وإعادة تشغيل الخادم — يتطلب منطق إعادة اتصال قويًا ومراقبة نبضات القلب ومخزنًا مؤقتًا لإعادة تشغيل الأحداث المفقودة
- وكيل Node.js المحلي مكوّن مخصص يجب إصدار إصداراته ونشره ومراقبته في كل عقار فندقي — النشرات متعددة العقارات تتطلب استراتيجية إدارة عن بعد
- أوامر الكتابة المتدفقة من الواجهة الخلفية MERN عبر WebSocket إلى Oracle المحلية تُدخل زمن تأخير كتابة ثنائي القفزة (MERN → WebSocket → الوكيل → Oracle) مقارنةً باتصال DB مباشر
- إذا انقطع اتصال WebSocket أثناء عملية كتابة، قد لا تتلقى الواجهة الخلفية MERN الإقرار — مفاتيح الأغراضية ومنطق إعادة المحاولة أساسيان لمنع الحجوزات المكررة
- عملية الوكيل تمثل اعتمادًا محليًا؛ فريق تقنية المعلومات في الفندق الذي يُعيد تشغيل الخادم دون إعادة تشغيل الوكيل سيكسر المزامنة بصمت حتى يُلاحَظ ذلك

### الحل الرابع: GraphQL Subscriptions + خدمات ويب OPERA (OWS) + MongoDB Atlas

**ملخص الهندسة المعمارية:**
يستهدف تثبيتات OPERA المحلية التي لا تتوفر فيها OHIP، ويستخدم هذا الحل **خدمات ويب OPERA (OWS)** — واجهة SOAP/XML الخاصة بـ Oracle لـ OPERA المحلي — كقناة كتابة، مُقدَّمة أمامها **طبقة GraphQL API** في الواجهة الخلفية MERN. يعرض خادم Node.js MERN مخطط GraphQL يُعيَّن مباشرةً إلى عمليات حجوزات OPERA. للقراءات، تُغذِّي **خدمة استطلاع OWS مجدوَلة** MongoDB Atlas بتردد عالٍ (كل 30–60 ثانية). للكتابات، تُشغِّل طفرات GraphQL استدعاءات SOAP المتزامنة لـ OWS إلى OPERA، مع استخدام الاستجابة لتأكيد الحجز قبل تحديث Atlas. يُستخدم عنوان IP الثابت للفندق لتقييد وصول OWS على خدمة السحابة حصريًا.

```
[React Frontend]
        │ GraphQL query / mutation / subscription
        ▼
[Node.js GraphQL Server (Apollo)]
        ├── Query ──────────────────────────────► [MongoDB Atlas]
        │                                               ▲
        │                                               │ OWS Sync (30s poll)
        └── Mutation ──► [OWS SOAP Client]──────────────┤
                                │                       │
                                ▼                       │
                    [OPERA Web Services]                │
                         (Static IP)                   │
                                │                       │
                                ▼                       │
                    [OPERA Oracle DB]───────────────────┘
                     INSERT/UPDATE on
                     reservation tables
```

**آلية المزامنة:** القراءات تُخدَم من Atlas (زمن تأخير منخفض، لا حمل على OPERA). الكتابات تذهب بشكل متزامن عبر OWS، مضمونةً أن OPERA هو نظام السجل. استطلاع OWS كل 30–60 ثانية يُبقي Atlas متسقًا مع حالات الوصول المباشر والتغييرات من نظام PMS. اشتراكات GraphQL (عبر Atlas Change Streams) تدفع التحديثات الفورية إلى عملاء React المتصلين.

**خريطة الامتثال:**
- ✅ TR-1: ثنائي الاتجاه — السحابة تستدعي OWS للكتابات؛ استطلاع عالي التردد للقراءات
- ✅ TR-2: دورة مزامنة OWS كل 30–60 ثانية تقترب من الفورية؛ الكتابات فورية الاتساق
- ✅ TR-3: عنوان IP ثابت مستخدم لإدراج IP الخدمة السحابية في القائمة البيضاء على جدار الحماية للفندق لمنفذ OWS
- ✅ SC-1: OWS عبر HTTPS (TLS 1.3)؛ Atlas TLS 1.3 أثناء النقل
- ✅ SC-2: PII مُعالَجة وفق متطلبات الحجوزات
- ✅ SC-3: بيانات اعتماد OWS محددة النطاق لعمليات الحجوزات؛ لا وصول Oracle مباشر من السحابة
- ✅ SA-1/SA-2: MFA على طبقة مصادقة GraphQL؛ Atlas لقراءات أصحاب المصلحة

**المكدس التقني:** Apollo GraphQL Server، خدمات ويب OPERA (SOAP)، node-soap، MongoDB Atlas، React، Atlas Change Streams  
**التعقيد:** ⭐⭐⭐ متوسط  
**ملف التكلفة:** منخفض-متوسط (OWS مُدرَج مع ترخيص OPERA المحلي؛ Atlas M10+)

**المزايا:**
- OWS متاح على تثبيتات OPERA المحلية دون اشتراك OHIP، مما يجعله قناة الكتابة الأكثر سهولة للفنادق غير المنتقلة بعد إلى OPERA Cloud
- المخطط المُكتَّب لـ GraphQL يوفر عقدًا نظيفًا وموثقًا ذاتيًا بين واجهة React الأمامية والواجهة الخلفية — طفرات `createReservation` و`updateReservation` صريحة ومُحددة الإصدار
- اشتراكات GraphQL مُدمَجة مع Atlas Change Streams تمنح واجهة React طبقة إشعارات دفع فورية بلا بنية تحتية إضافية
- OWS كان الآلية القياسية لتكامل OPERA للنشرات المحلية لسنوات عديدة — موثق جيدًا ومعتمد من دعم Oracle، مما يُقلل مخاطر التنفيذ
- Atlas يمتص جميع حركة القراءة مما يعني ضرب خادم OPERA فقط للكتابات واستطلاع المزامنة، مما يُقلل الحمل بشكل كبير على نظام PMS الإنتاجي

**العيوب:**
- OWS يستخدم SOAP/XML الذي يتسم بالإسهاب وبطء التحليل وصعوبة التعامل معه من قِبَل المطورين مقارنةً بـ REST أو GraphQL — مكتبة `node-soap` في Node.js تُضيف تعقيدًا وتعيين استجابة XML يتطلب صيانة دقيقة
- دورة الاستطلاع كل 30–60 ثانية تعني أن Atlas يمكن أن يتخلف عن OPERA بمقدار دقيقة للتغييرات من نظام PMS — قد يكون هذا مقبولًا لمعظم القراءات لكنه قد يسبب تناقضات توافر مؤقتة مرئية لمستخدمي الحجوزات
- متطلب عنوان IP الثابت على جدار الحماية في الفندق يعني تنسيق قاعدة فتح منفذ مع فريق تقنية المعلومات لمنفذ HTTP الخاص بـ OWS — هذا قابل للإدارة لكنه يتطلب عمليات التحكم في التغييرات في كل عقار
- مصادقة OWS تستخدم نمط WSSE username/token الذي يتطلب إدارة دقيقة لتدوير بيانات الاعتماد في الواجهة الخلفية MERN
- تغطية ميزات OWS تتفاوت بحسب إصدار OPERA؛ ميزات الحجوزات المتقدمة (الباقات والملفات الشخصية المرتبطة والتخصيصات) قد تتطلب سلاسل استدعاء SOAP متعددة لتنفيذ عملية منطقية واحدة

### الحل الخامس: مزامنة مُشغَّلة بالأحداث عبر Oracle GoldenGate / Debezium CDC + ناقل أحداث MERN + كتابات OHIP

**ملخص الهندسة المعمارية:**
الخيار الأكثر قوةً ومستوى المؤسسة. يبثّ **التقاط تغييرات البيانات (CDC)** عبر Oracle GoldenGate (أو Debezium مفتوح المصدر مع Oracle LogMiner) كل تغيير مُلتزَم به في OPERA Oracle DB — على مستوى سجل الإعادة — إلى **ناقل أحداث Kafka أو AWS MSK** في السحابة. تستهلك الواجهة الخلفية Node.js في MERN هذه الأحداث وتُطبِّقها على MongoDB Atlas في غضون ميلي ثوانٍ، محققةً مزامنة مستمرة حقيقية وفق TR-2 دون أي استطلاع. تتدفق عمليات الكتابة من الواجهة الخلفية MERN عبر **OHIP REST API** (لـ OPERA Cloud) أو **OWS** (لـ OPERA المحلي)، موفرةً قناة كتابة نظيفة مُتحقَّق منها من Oracle. يُستخدم عنوان IP الثابت للفندق لتأمين نقطة نهاية نسخ GoldenGate.

```
[OPERA Oracle DB]
        │ Redo log capture
        ▼
[GoldenGate / Debezium CDC Agent]
        │ TLS 1.3 — Static IP outbound
        ▼
[Kafka / AWS MSK Event Bus]
        │
        ├──► [MERN Node.js Consumer] ──► [MongoDB Atlas]
        │                                      │
        │                              [React Frontend]
        │                              [Stakeholder Dashboard]
        │
        └──► [Audit / Analytics Pipeline]

[MERN Write Path]
[React Frontend] ──► [Node.js Backend] ──► [OHIP / OWS] ──► [OPERA Oracle DB]
                                                │
                                        (Confirmation event
                                         re-enters Kafka via CDC)
```

**آلية المزامنة:** CDC هو المعيار الذهبي للمزامنة المستمرة — كل INSERT وUPDATE ومعاملة مُلتزَم بها في OPERA تُلتقط على مستوى محرك قاعدة البيانات وتُنشر إلى Kafka في غضون ميلي ثوانٍ. Atlas يُبقى حديثًا على الدوام. الكتابات من الواجهة الخلفية MERN تذهب عبر OHIP/OWS، وبمجرد الالتزام بها في Oracle، تُلتقط تلقائيًا بواسطة CDC وتنعكس في Atlas — مُنشئةً حلقةً مغلقة متسقة ذاتيًا.

**خريطة الامتثال:**
- ✅ TR-1: ثنائي الاتجاه — CDC يدفع تغييرات OPERA للخارج؛ OHIP/OWS يتعامل مع الكتابات من السحابة إلى OPERA
- ✅ TR-2: CDC يوفر مزامنة مستمرة على مستوى الميلي ثانية — أقوى امتثال لـ TR-2 من أي حل
- ✅ TR-3: عنوان IP ثابت مستخدم لتأمين نقطة نهاية هدف نسخ GoldenGate؛ لا منفذ DB خام مكشوف
- ✅ SC-1: GoldenGate TLS 1.3؛ Kafka TLS 1.3؛ OHIP TLS 1.3؛ Atlas TLS 1.3 — السلسلة الكاملة مفروضة
- ✅ SC-2: حقول PII يمكن إخفاؤها في طبقة تحويل CDC قبل وصول الأحداث إلى Kafka
- ✅ SC-3: وكيل CDC يستخدم مستخدم Oracle مخصص لاستخراج السجلات؛ OHIP/OWS يتعامل مع الكتابات دون وصول DDL مباشر
- ✅ SA-1/SA-2: MFA على طبقة مصادقة MERN؛ قوائم التحكم في الوصول (ACLs) لـ Atlas وKafka تُقيِّد وصول المستهلكين

**المكدس التقني:** Oracle GoldenGate أو Debezium، Apache Kafka / AWS MSK، OHIP أو OWS، Node.js، MongoDB Atlas، React  
**التعقيد:** ⭐⭐⭐⭐⭐ مرتفع  
**ملف التكلفة:** مرتفع (ترخيص GoldenGate + مجموعة Kafka + OHIP إذا كان قابلًا للتطبيق)

**المزايا:**
- الحل الوحيد الذي يحقق مزامنة حقيقية مستمرة دون الثانية من OPERA إلى Atlas دون أي استطلاع — CDC على مستوى سجل الإعادة يلتقط كل معاملة مُلتزَم بها، بما فيها التغييرات من نظام PMS والوصول المباشر والتعديلات المكتبية الخلفية التي لن تُظهرها أي API
- Kafka كناقل أحداث يفصل OPERA عن الواجهة الخلفية MERN كليًا — لا يحتاج نظام PMS أن يعرف بوجود SaaS؛ كل منطق التكامل يقطن في المستهلكين
- إعادة تشغيل الأحداث: إذا تعطلت الواجهة الخلفية MERN أو Atlas، يحتفظ Kafka بسجل الأحداث ويمكن للمستهلكين إعادة التشغيل من آخر إزاحة مُلتزَم بها — فقدان صفري للبيانات
- مسار الكتابة (OHIP/OWS) ومسار القراءة (CDC) مستقلان تمامًا — توقف API الكتابة لا يؤثر على مزامنة القراءة، والعكس صحيح
- قابل للتوسع بشكل كبير: يمكن إضافة مستهلكين إضافيين على ناقل Kafka للتحليلات والتقارير وإدارة الإيرادات والتكاملات مع أطراف ثالثة دون تعديل OPERA أو الواجهة الخلفية MERN

**العيوب:**
- الأغلى والأكثر تعقيدًا تشغيليًا في هذا المستند؛ ترخيص Oracle GoldenGate بأسعار مؤسسية والخبرة التشغيلية لتشغيله نادرة
- Debezium مع Oracle LogMiner هو البديل مفتوح المصدر لكنه يتطلب ضبطًا دقيقًا — لـ LogMiner تداعيات أداء معروفة على قواعد بيانات Oracle عالية المعاملات ويمكن أن يؤثر على حمل خادم OPERA إذا لم يُهيَّأ بشكل صحيح
- يتطلب Kafka نشره ومراقبته وإدارة الموضوعات وإدارة مجموعات المستهلكين وسياسة الاحتفاظ — التزام تشغيلي غير تافه فوق مكدس MERN الحالي
- مزيج CDC + Kafka + OHIP + Atlas هو أكبر سطح في هذا المستند؛ تصحيح تناقض مزامنة يتطلب تتبع حدث عبر أربعة أنظمة متمايزة
- الإعداد الأولي ينطوي على مشاركة Oracle DBA بشكل كبير لتهيئة التسجيل التكميلي وعمليات استخراج GoldenGate وملفات المسار — هذا ليس نشرًا ذاتي الخدمة

## 4. الهندسة الموصى بها <a name="recommendation"></a>

### التوصية الأولية: الحل الثالث (وكيل Node.js المحلي + WebSocket)

لفريق MERN الذي يبني منتج SaaS جديدًا، يُقدم **الحل الثالث** أفضل توازن بين أداء المزامنة الفورية وقابلية الكتابة والبساطة التشغيلية:

- الوكيل مكتوب بنفس لغة بقية المكدس — لا Oracle DBA مطلوب، لا مكتبات SOAP، لا ترخيص منفصل
- WebSocket ثنائي الاتجاه الفوري يستوفي TR-2 باستمرار دون تأخير الاستطلاع
- بيانات اعتماد Oracle لا تغادر شبكة الفندق أبدًا — الخدمة السحابية معزولة عن وصول DB المباشر
- عنوان IP الثابت متاح إذا احتُيج إليه لكن الهندسة لا تعتمد عليه، مما يُبسِّط النشر متعدد العقارات

### التوصية الثانوية: الحل الأول (OHIP ثنائي الاتجاه)

إذا كانت مجموعة الفنادق على **OPERA Cloud** والميزانية تدعم ترخيص OHIP، فالحل الأول هو الخيار الأنظف معماريًا. OHIP يتعامل مع جميع تعقيدات Oracle، التحقق من الكتابة مُفعَّل من Oracle، والواجهة الخلفية MERN لا تحتاج أبدًا إلى مكتبة عميل Oracle.

### للحصول على أقصى دقة في المزامنة: الحل الخامس (CDC + Kafka)

إذا كان المتطلب غير القابل للتفاوض هو **عدم تحمل أي تأخير في المزامنة** عبر عقارات كثيرة على نطاق واسع، فالحل الخامس هو الهندسة الوحيدة التي تُوصِّل مزامنة مستمرة على مستوى الميلي ثانية. التكلفة والتعقيد مبررَان فقط على نطاق المؤسسة (10+ عقارات) أو حيث تعتمد المعاملات المالية على بيانات التوافر الفورية.

### ما يجب تجنبه

يجب التعامل مع **الحل الثاني (Oracle مباشر + VPN)** باعتباره الملاذ الأخير. رغم أنه يُقدم أقصى مرونة في الاستعلامات، فإن وضع بيانات اعتماد Oracle المباشرة للكتابة في البيئة السحابية يُنشئ خطرًا غير مقبول على قاعدة بيانات OPERA الإنتاجية. إذا اختير، يجب إقرانه بجدار حماية تطبيقات الويب (WAF) وبروكسي الاتصالات وسياسة صارمة لتدوير بيانات الاعتماد.
